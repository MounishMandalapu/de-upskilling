## Answer Key
*(Review after answering — be honest!)*

**A1.** A data engineer builds and maintains the systems that collect, store, move, and make data reliably available for analysts, scientists, and applications.

**A2.** ETL: Extract data from source → Transform it (clean/reshape) → Load into destination. Used when your destination can't handle transformations (e.g., loading into a simple DB). ELT: Extract → Load raw data first into a powerful warehouse → Transform inside the warehouse. Used with Snowflake/BigQuery which have massive compute — dbt follows this pattern.

**A3.** Airflow = orchestrator (schedules/monitors pipelines) | dbt = SQL transformation layer in warehouse | Kafka = real-time event streaming | Spark = distributed large-scale data processing | Snowflake = cloud data warehouse (storage + analytics)

**A4.** Apache Spark / PySpark. It's designed specifically for distributed processing of massive datasets — it splits the 500M rows across many machines and processes them in parallel. A single Python script or SQL query would run out of memory.

**A5.** Apache Kafka. It's a real-time event streaming platform designed to handle high-volume events (like purchase events) with millisecond latency. A consumer reading from Kafka could trigger the discount within 2 seconds.

**A6.** A DAG is a Directed Acyclic Graph — a set of tasks connected by dependencies, where the flow is one-directional and there are no circular loops. Airflow uses DAGs because pipelines have dependencies (e.g., "load data before transforming it") and circular dependencies would cause infinite loops.

**A7.** False. A data lake stores raw, unprocessed data in any format (cheap storage, like S3). A data warehouse stores structured, clean data optimized for analytical queries (like Snowflake). A data lakehouse (Delta Lake, Iceberg) tries to combine both.

## Bonus Answer Key

**AB1.** Idempotency means running a pipeline multiple times produces the same result as running it once. Critical because pipelines fail and get re-run all the time. Example of what breaks: a non-idempotent pipeline appends rows on every run — if it fails halfway and you re-run, you get duplicate rows in the warehouse. Fix: use MERGE/UPSERT instead of INSERT, or truncate-and-reload with a transaction.

**AB2.** Root causes to investigate: (1) **Silent data corruption upstream** — a source table had a schema change or had its data modified after the pipeline ran; check row counts and checksums against the source. (2) **Incorrect join/aggregation logic** — the pipeline ran successfully but a JOIN was producing a cartesian product or incorrect grain; check intermediate tables and row counts at each stage. (3) **Timezone/date boundary issue** — the pipeline ran at 2am but was filtering on "today" which resolved differently than expected; check the date filter logic and compare pipeline run time vs. data timestamps.
